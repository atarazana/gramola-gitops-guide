Prepare Repositories

Source Of Truth

As we have already exposed in GitOps Git is the source of truth so we need one to start with…​ you may have your own but for the sake of homogeneity in this guide we’re going to use a git server deployed in OpenShift.

Why not relying on a preexisting git server somewhere you may ask? This is so, because in order to execute the pipelines section of this guide you have to fork some git repositories.

This is necessary because:

  • webhooks need to be created and obviously you need permissions on them

  • you want to make changes to the code and promote those changes across environments

These are the repositories you need to fork:

  • Configuration Repo: https://github.com/atarazana/gramola.git

  • Events Microservice: https://github.com/atarazana/gramola-events.git

  • Gateway Microservice: https://github.com/atarazana/gramola-gateway.git

You could fork them in your preferred git server, but for the sake of simplicity we’re going to give you instructions to deploy a git server in OpenShift and for them there.

Deploying Gitea

Run this command to deploy the Gitea operator and Gitea itself in your OpenShift cluster.

Gitea will be installed in gitea-system namespace 
until kubectl apply -k util/gitea/; do sleep 2; done

The previous command will retry until it’s all installed but we still need to check if it all works properly.

Let’s have a look to the Gitea deployment object.

This command only works as expected once the Deployment object has been created by the operator, be patient and try again if you get.

Error from server (NotFound): deployments.apps "repository" not found
 
kubectl rollout status deploy/repository -n gitea-system

This is the normal state after being successfully rolled out.

deployment "repository" successfully rolled out
get deployment/repository -n gitea-system
NAME         READY   UP-TO-DATE   AVAILABLE   AGE
repository   1/1     1            1           32m

Once Gitea has been rolled out, run this little smoke test.

curl -I https://repository-gitea-system.apps.%BASE_SUBDOMAIN%/
You should get something like this. If you don’t, be patient and run this command in 2 mins or so. 
HTTP/1.1 200 OK
set-cookie: i_like_gitea=82e42fd387b6ea7a; Path=/; HttpOnly; SameSite=Lax
date: Mon, 25 Oct 2021 08:06:22 GMT
set-cookie: 5b3fc896c2f6eedc568a6b59cd1862ea=23d42cb20150039ef52136e8993c6865; path=/; HttpOnly; Secure; SameSite=None
cache-control: private

Create Git Users

Now that Gitea is installed and up and running, it’s time to some users in the cluster, to do so we’re going to create a Job that will in it’s turn create 20 users.

kustomize build util/setup/ | sed -e 's/^  name\:.*$//g' | kubectl create -f -

If you just want to create one use do this.

Ignore error "Internal Server Error" user %USERNAME% should have been created. 
GITEA_INSTALL_TOKEN=$(curl -s -XPOST -H "Content-Type: application/json"  -k -d '{"name":"install"}' -u opentlc-mgr:r3dh4t1\! https://repository-gitea-system.apps.%BASE_SUBDOMAIN%/api/v1/users/opentlc-mgr/tokens | jq -r .sha1)
echo "GITEA_INSTALL_TOKEN=${GITEA_INSTALL_TOKEN}"

curl -X 'POST' \
    "https://repository-gitea-system.apps.%BASE_SUBDOMAIN%/api/v1/admin/users?access_token=${GITEA_INSTALL_TOKEN}" \
    -H 'accept: application/json' \
    -H 'Content-Type: application/json' \
    -d '{
          "email": "%USERNAME%@example.com",
          "full_name": "User %USERNAME%",
          "login_name": "%USERNAME%",
          "must_change_password": false,
          "password": "openshift",
          "send_notify": true,
          "source_id": 0,
          "username": "%USERNAME%",
          "visibility" : "public"
    }'

Explore Gitea Console

Before we continue with the migration of the git repositories of our application open the console and check that your credentials are valid and all looks good. Open https://repository-gitea-system.apps.%BASE_SUBDOMAIN% and use these credentials:

  • USERNAME: %USERNAME%

  • PASSWORD: openshift

You should see something like this. Please enter your credentials and hit the green button.

Create Git PAT

Let’s create a Personal Access Token (PAT) in Gitea for user %USERNAME%.

You’ll need this token later so don’t forget to copy it somewhere just in case you close the terminal window. 
GIT_PAT=$(curl -s -XPOST -H "Content-Type: application/json"  -k -d '{"name":"cicd'"${RANDOM}"'"}' -u %USERNAME%:openshift https://repository-gitea-system.apps.%BASE_SUBDOMAIN%/api/v1/users/opentlc-mgr/tokens | jq -r .sha1)
echo "GIT_PAT=${GIT_PAT}"

Migrate Repositories

Now that user %USERNAME% has been created let’s migrate gramola repositories into Gitea.

  curl -X 'POST' \
      "https://repository-gitea-system.apps.%BASE_SUBDOMAIN%/api/v1/repos/migrate" \
      -H 'accept: application/json' \
      -H  "Authorization: token ${GIT_PAT}" \
      -H 'Content-Type: application/json' \
      -d '{
      "clone_addr": "https://github.com/atarazana/gramola.git",
      "description": "gramola configuration",
      "issues": false,
